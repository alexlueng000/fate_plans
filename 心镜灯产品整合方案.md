## 一、产品评估

### 1.1 两份文档对比

| 维度 | 心迹.md（MVP） | 心迹2.md（完整版） |
|------|---------------|-------------------|
| 功能模块 | 8个独立模块 | 5大模块（整合优化） |
| 核心创新 | ACT+叙事治疗 | **节气+八字+心理学** |
| 目标用户 | 心理健康意识用户 | 中国文化认同用户 |
| 差异化 | 中等 | **极高（文化护城河）** |
| 与fate协同 | 无 | **天然协同** |

**优先实施：心迹2.md（心镜灯完整版）**

1. 24节气情绪归因系统与fate的八字系统形成完美互补
2. 独特的文化融合难以被竞品复制
3. 可直接复用fate的用户系统和八字计算引擎
4. 形成"算命→疗愈"的完整闭环，提升用户LTV

## 三、界面流程优化

### 3.1 核心痛点

原设计存在的问题：
1. ✅ 功能太多（8个模块），用户不知道从哪开始
2. ✅ 情绪记录流程太长（3个独立页面）
3. ✅ 深度功能门槛太高（7个问题一次性完成）
4. ✅ 缺少即时反馈（记录完才有反馈）

### 3.2 优化方案："一次对话"流程

#### 优化前（3个独立页面）
```
镜厅 → 心迹 → 身体感受（保存）→ 返回
              → 来了什么（保存）→ 返回
              → 它想说什么（保存）→ 返回
```

#### 优化后（1个连贯对话流）
```
镜厅 → 点击"今天，和自己说说话"
       ↓
   [卡片式对话流]
   
   步骤1："身体哪里有感觉？"
   [胸口] [肩膀] [胃] [跳过]
   → 点选后立即反馈："肩膀感觉到了。"
   
   步骤2："今天有谁来了？"
   😰恐惧 😤烦躁 😞内疚 [跳过]
   → 点选后立即反馈："恐惧来了。它来的时候，肩膀感觉到了。"
   → 自动关联步骤1的身体部位
   
   步骤3："它想说什么？"
   [文本框] [跳过]
   → 输入后显示翻译："那不是你，是心里一个声音..."
   
   步骤4：即时反馈
   "记下了。今日谷雨。有些感觉来了，也可以走。"
   + 显示本周小图
```

**关键改进**：
- 从3个页面 → 1个对话流（减少70%跳转）
- 每步都有即时反馈
- 自动关联上下文
- 可随时跳过

**时间成本**：
- 新用户首次：30秒（vs 原设计2-3分钟）
- 老用户日常：15秒

### 3.3 首页简化设计

```
┌─────────────────────────────┐
│  今日谷雨 · 傍晚              │ ← 节气+时辰
│  有些感觉来了，也可以走。      │ ← 引导语
├─────────────────────────────┤
│                             │
│   [今天，和自己说说话]        │ ← 主按钮（唯一动作）
│                             │
├─────────────────────────────┤
│  你像一棵树。                 │ ← 天性小像
│  想向上长，也会被风压弯。      │
├─────────────────────────────┤
│  本周小记 ✨✨               │ ← 本周数据预览
│  [查看详情]                  │
└─────────────────────────────┘
```

**设计原则**：
- 一个明确的主动作
- 本周数据前置（即时价值感）
- 减少认知负担

### 3.4 深度功能降低门槛

#### 优化前
```
夜灯 → 走一段路 → 7个问题必须答完
```

#### 优化后（渐进式深度）
```
夜灯 → "今晚想聊什么？"
       ├─ 快速版（3问）← 新增，降低门槛
       ├─ 完整版（7问）← 原有流程
       └─ 自由写 ← 零门槛
```

**触发时机**：
- 不在首次使用时推荐
- 等用户记录3天后再引导
- 用"解锁"心理降低抗拒

        emotion_type=emotion_type,

    db.commit()
    
    return {

        "feedback": get_feedback_message(solar_term, time_period)
    }

# 2. 获取本周数据
@router.get("/weekly-chart")
async def get_weekly_chart(user = Depends(get_current_user)):
    """获取本周情绪图表数据"""
    week_start = date.today() - timedelta(days=date.today().weekday())
    records = db.query(EmotionRecord).filter(
        EmotionRecord.user_id == user.id,
        EmotionRecord.record_date >= week_start
    ).all()
    
    return {
        "week_start": week_start,
        "records": [r.to_dict() for r in records],
        "exception_moments": get_exception_moments(user.id, week_start)
    }

# 3. 获取性格画像（基于八字）
@router.get("/character")
async def get_character_profile(user = Depends(get_current_user)):
    """获取天性小像（基于用户的八字数据）"""
    if not user.default_birth_data:
        # 无八字数据，使用季节版本

    # 有八字数据，使用命理版本

    paipan = calc_paipan(**birth_data)
    wuxing_profile = analyze_wuxing(paipan)
    
    return {
        "type": "bazi",
        "profile": wuxing_profile,
        "description": get_wuxing_description(wuxing_profile)
    }

### 4.9 需要解决的矛盾

#### 矛盾1：数据本地化 vs 跨设备同步

**原设计承诺**："所有记录只留在你这里"（本地存储）  
**实际情况**：fate是Web应用，用户期望跨设备访问

**解决方案**：
- 数据存储在服务器（加密）
- 文案调整为："所有记录只有你能看到，不会被分享或分析"
- 提供数据导出功能（JSON）
- 提供一键删除功能

#### 矛盾2：命理版 vs 非命理版

**原设计**：两个版本供用户选择  
**实际情况**：fate用户都有八字数据

**解决方案**：
- 默认使用命理版（用户已接受八字概念）
- 在设置中提供"简化模式"切换（隐藏命理术语）
- 新用户（无八字数据）自动使用非命理版
