# 用户档案系统重构 - 工作量评估

## 项目范围

将现有"临时排盘"模式升级为"持久化用户档案"系统，实现：
- 一个账号一个默认命盘档案
- 所有聊天会话自动绑定命盘
- 修改资料后自动重算命盘
- 完整的状态机路由（未登录 → 建档 → 聊天）

**不包括**：小程序（已废弃）、多档案切换、合盘功能

## 关键风险点

### 1. 老用户数据迁移
**风险**：现有聊天记录如何关联到新命盘？
**方案**：
- 历史会话标记为"未绑定命盘"
- 首次登录后引导建档
- 新会话自动绑定新命盘

### 2. 状态机边界判断
**风险**：登录态 + 建档态的组合判断容易出 bug
**方案**：
- 在 middleware 层统一处理
- 编写完整的单元测试覆盖所有状态组合

### 3. 表单输入体验
**风险**：日期/时间选择器在不同浏览器表现不一致
**方案**：
- 首版使用 HTML5 原生组件（简化开发）
- 后续可升级为第三方组件库（如 react-datepicker）

### 4. 自动重算性能
**风险**：用户频繁修改资料导致重复计算
**方案**：
- 前端防抖（debounce）
- 后端缓存命盘结果

## 结论

这是一次**中大型重构**，核心是把"临时排盘"变成"持久化档案"，影响产品的根本使用逻辑。

**建议执行方式**：
1. 按阶段顺序推进，不要跳跃
2. 每个阶段完成后立即测试，不要积累问题
3. 预留 20% buffer 处理意外情况
4. 优先保证核心链路稳定，辅助功能可后续迭代

**预期收益**：
- 用户留存率提升（有档案后更容易回访）
- 对话质量提升（AI 基于固定命盘分析更连贯）
- 商业化基础（档案是会员体系的前置条件）
