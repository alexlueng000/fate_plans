# 评估：个人数据仪表盘功能

## 功能描述
向用户展示使用统计："你已进行15次对话，提出47个问题，获得89条洞察"  
进度追踪："你的人生清晰度评分：72/100"（鼓励更多使用）

---

## 一、商业价值评估

### 高价值点 ✅
- **留存钩子**：数字化的"历史"让用户产生沉没成本感，不舍得离开
- **使用激励**：进度分数天然引导用户多次回来提问，提升 DAU/MAU
- **情感连接**："你已陪伴我走过15次对话"比冷冰冰的续费提醒更温暖
- **差异化**：竞品几乎没有做使用数据反馈，有稀缺性

### 需要警惕的点 ⚠️
- **"人生清晰度评分"是虚构指标**：算法设计必须让用户觉得"合理"而非"滑稽"，否则会破坏信任感
- **低频用户可能反效果**："你只提了3个问题" 对轻度用户是负向激励
- **数字太小会尴尬**：新用户看到 0 条洞察，仪表盘显得空洞

### 价值结论
**中高价值**——对高频/付费用户意义显著，但需要处理好新用户的空状态体验。

## 二、实现难度评估

### 后端现状（fate/）
**✅ 数据已经存在，无需重新采集：**
- `conversations` 表：每次会话有记录 → 对话次数
- `messages` 表：`role='user'` 的行数 → 问题数量
- `messages` 表：`role='assistant'` 的行数 → 洞察数量（可计为 AI 回复数）
- `usage_logs` 表：已记录 `chat_start` / `chat_send` 事件

**需要新增：**
1. 一个统计聚合接口 `GET /api/user/stats`，返回对话数/问题数/洞察数
2. "人生清晰度评分"算法（见下方）
3. 可选：在 `users` 表加冗余计数字段（避免每次实时 COUNT，数据量大时有用）

**关键文件：**
- `fate/app/models/chat.py` — Conversation & Message 模型
- `fate/app/models/usage_log.py` — UsageLog 模型
- `fate/app/routers/` — 需新增 stats router

### 前端现状（fate-frontend/）
**✅ UI 组件已就绪，几乎可以直接复用：**
- `app/components/charts/MetricCard.tsx` — 现成的统计数字卡片（含趋势箭头）
- `app/components/dashboard/Dashboard.tsx` — 仪表盘容器已存在
- `app/components/ui/` — Card、Tabs、Skeleton 全套组件
- `app/account/page.tsx` — 用户中心，天然的挂载点

**需要新增：**
1. 调用 `/api/user/stats` 的前端逻辑
2. 在 `account/page.tsx` 或新页面展示统计卡片
3. 进度条 / 评分展示组件（可能需要新建，但基于现有 Card 组件很快）

## 三、"人生清晰度评分"算法设计建议

这是功能中最敏感的部分，需要让用户觉得"合理"：

```
基础分：50分（所有人起点）
+ 对话次数贡献：min(对话数 × 3, 20) 分  →  最多 +20
+ 提问多样性贡献：涉及话题种类 × 2  →  最多 +15
+ 近期活跃度：最近30天有对话 +10，最近7天 +5
+ 洞察深度：平均每次对话消息轮数 > 5 +5
= 总分上限 100
```

**注意**：分数要让用户"永远还有提升空间"，避免满分，建议上限设计为 85-90 分。

## 五、建议的实现优先级

### Phase 1（1天，高 ROI）
- 后端：`GET /api/user/stats` 返回对话数、问题数、洞察数
- 前端：在 `account/page.tsx` 用 MetricCard 展示三个数字

### Phase 2（1天，增加黏性）
- 实现"人生清晰度评分"算法
- 前端加进度条展示，配上鼓励文案

### Phase 3（可选，锦上添花）
- 趋势图：最近30天提问频次折线图
- 话题词云：用户最常问的关键词
- 徽章系统："初窥门径"、"探寻者"、"命理达人"

---

## 六、结论

**值得做，而且容易做。**

基础设施（数据、UI组件）已经完备，Phase 1 是一个典型的"低投入高回报"功能。  
唯一需要谨慎的是"清晰度评分"的措辞和算法——建议内部测试后再上线，确保不会让用户觉得荒谬。
