产品主逻辑

对用户来说，流程应该变成这样：

未登录用户

先看到价值页
→ 点击“立即开始分析”
→ 弹出登录
→ 登录成功后继续下一步

已登录但未建档用户

→ 填出生信息
→ 生成命盘并保存

已登录且已建档用户

直接进入聊天首页
→ 默认基于已保存命盘继续提问

修改资料用户

进入资料编辑页
→ 保存
→ 系统自动重算命盘
→ 后续聊天基于新命盘

二、方案一的正式边界

这版只做一个账号一个默认命盘档案，但底层按多档案预埋。

这版做什么
必须登录
一个账号仅 1 个默认命盘档案
首次建档排盘一次
后续直接进入聊天
修改资料后自动重算
所有聊天会话绑定当前命盘
这版不做什么
多档案切换
合盘
家庭成员管理
多命盘并行会话
时间校准高级模式
太复杂的会员体系

这很重要。
你现在要的是把主链路打通，不是把全部未来功能一次性做完。

三、最关键的状态机

这是整个产品体验是否顺的根本。

if not logged_in:
    show landing_page
elif logged_in and no_profile:
    show onboarding_build_profile
elif logged_in and has_profile:
    show chat_home

再补一条：

if profile_updated and input_changed:
    recalculate_chart
    bind new chart to new sessions

也就是说，用户不是在“访问页面”，而是在系统状态里被自动送到最合理的下一步。

七、响应式设计原则

所有页面采用响应式设计，移动端优先。

桌面端适配建议
未登录落地页：居中布局，最大宽度 1200px
首次建档页：左右布局（左表单右价值说明），移动端垂直堆叠
聊天首页：最大宽度 800px 居中，保持聊天体验
查看命盘页：卡片式布局，桌面端可展示更多细节
修改资料页：同首次建档页

重点是：
所有端遵守同一套状态机逻辑
响应式布局而非多套代码

八、技术实现要点

前端路由守卫
在 Next.js middleware 或 layout 层实现状态机：
- 检查登录态（JWT token）
- 检查命盘档案（API 调用 /api/auth/me）
- 根据状态自动重定向

后端数据模型
新增表：user_profiles（用户档案）
- user_id（外键关联 users 表）
- gender, calendar_type, birth_date, birth_time, birth_location
- bazi_chart（JSON 存储命盘数据）
- created_at, updated_at

修改表：chat_sessions
- 新增 profile_id（外键关联 user_profiles）
- 所有会话必须绑定命盘

自动重算逻辑
用户修改资料 → 触发后端 recalculate_chart()
→ 更新 user_profiles.bazi_chart
→ 后续新会话自动使用新命盘

九、数据迁移策略

老用户处理
- 已有聊天记录的用户：首次登录后引导建档
- 建档完成后，历史会话标记为"未绑定命盘"（可选择性关联）
- 新会话自动绑定新命盘
