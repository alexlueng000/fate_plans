# 网站真实接入微信支付与支付宝支付实施文档

## 一、项目背景

当前网站已经具备模拟支付、商品套餐、订单、支付记录、权益发放和次数扣减的基础能力。现阶段目标是将模拟支付升级为真实微信支付和支付宝支付，使用户可以在网站内购买八字、六爻等服务次数，并在支付成功后自动到账。

本项目的核心不是简单替换一个支付接口，而是建立一套稳定、可追踪、可对账、可处理异常的真实支付闭环。

真实支付上线后，必须解决以下问题：

1. 用户能清楚知道购买什么、支付多少钱、获得多少权益。
2. 用户支付成功后，系统能自动发放对应次数。
3. 用户支付后未到账时，系统能查单和补发。
4. 支付渠道重复回调时，不会重复发放权益。
5. 支付失败、取消、超时、退款等状态都能被正确处理。
6. 后台可以查看订单、支付流水、权益发放和异常记录。

---

## 二、总体目标

第一阶段目标：

- 支持真实支付宝网页支付。
- 支持真实微信扫码支付。
- 用户购买套餐后自动获得八字和六爻次数。
- 支付成功后前端显示到账结果。
- 次数用完后引导用户购买。
- 后端具备订单、支付流水、回调、查单、幂等、权益发放能力。

第二阶段目标：

- 支持手机浏览器支付。
- 支持微信 H5 支付。
- 支持支付宝手机网站支付。
- 增加订单记录页和简单后台管理能力。

第三阶段目标：

- 支持微信内置浏览器 JSAPI 支付。
- 支持退款、对账、异常订单处理后台。
- 优化转化路径和付费体验。

## 三、需要准备的资质与材料

### 3.1 微信支付准备项

接入微信支付前，需要准备：

1. 微信支付商户号。
2. 营业执照。
3. 法人或超级管理员身份信息。
4. 对公结算账户或允许绑定的结算银行卡。
5. 经营类目。
6. 客服电话。
7. 网站域名。
8. ICP 备案。
9. 微信支付产品权限，例如 Native 支付、H5 支付、JSAPI 支付。
10. 微信支付商户 API 证书。
11. APIv3 密钥。
12. 商户私钥。
13. 商户证书序列号。
14. 微信支付回调通知地址。

需要特别注意：

- 微信支付密钥、证书、私钥只能保存在后端服务器。
- 不得写入前端代码。
- 不得提交到 Git 仓库。
- 生产环境和测试环境必须分开配置。

### 3.2 支付宝支付准备项

接入支付宝前，需要准备：

1. 企业支付宝账号或商家账号。
2. 支付宝开放平台账号。
3. 支付宝开放平台应用。
4. APPID。
5. 应用私钥。
6. 应用公钥。
7. 支付宝公钥或支付宝公钥证书。
8. 电脑网站支付产品权限。
9. 手机网站支付产品权限。
10. 异步通知地址 notify_url。
11. 同步返回地址 return_url。
12. 网站域名。
13. ICP 备案。
14. 营业执照。
15. 经营类目。

需要特别注意：

- 支付宝同步跳转 return_url 不能作为支付成功依据。
- 真正的支付成功必须以后端异步通知验签成功，或主动查单确认为准。
- 支付宝私钥只能放在后端。

## 四、支付产品选择建议

### 4.1 PC 网页端

PC 端建议接入：

| 支付渠道 | 推荐产品 | 用户体验 |
|---|---|---|
| 微信支付 | Native 扫码支付 | 网站生成二维码，用户用微信扫码付款 |
| 支付宝 | 电脑网站支付 | 跳转支付宝收银台完成付款 |

PC 端第一版优先实现这两个即可。

### 4.2 手机浏览器端

手机浏览器建议接入：

| 支付渠道 | 推荐产品 | 用户体验 |
|---|---|---|
| 微信支付 | H5 支付 | 跳转微信支付页面或调起微信支付 |
| 支付宝 | 手机网站支付 | 调起支付宝 App 或进入支付宝网页收银台 |

### 4.3 微信内置浏览器

如果用户从微信聊天、公众号、朋友圈、微信群中打开网站，后续应接入微信 JSAPI 支付。

但 JSAPI 支付涉及公众号、小程序、openid、OAuth、APPID 绑定等额外配置，不建议第一阶段就做。

建议顺序：

1. 支付宝 PC / WAP。
2. 微信 Native 扫码支付。
3. 微信 H5 支付。
4. 微信 JSAPI 支付。

## 五、前端页面设计

真实支付上线后，前端不能只有一个“立即购买”按钮。至少需要完整覆盖购买、确认、支付、等待、成功、失败、订单记录等路径。

### 5.1 定价页

路径建议：

```txt
/pricing
```

页面应包含：

1. 当前用户剩余次数。
2. 套餐列表。
3. 每个套餐包含的权益。
4. 套餐价格。
5. 套餐有效期。
6. 是否推荐。
7. 支付安全说明。
8. 退款说明。
9. 常见问题。

建议套餐结构：

| 套餐 | 价格 | 权益 | 定位 |
|---|---:|---|---|
| 体验包 | ¥19 | 5 次八字 + 1 次六爻 | 降低首次付费门槛 |
| 基础套餐 | ¥99 | 10 次八字 + 3 次六爻 | 主推轻度用户 |
| 进阶套餐 | ¥299 | 50 次八字 + 15 次六爻 | 面向深度用户 |

如果暂时只保留现有两个套餐，也可以：

| 套餐 | 价格 | 权益 |
|---|---:|---|
| 基础套餐 | ¥99 | 10 次八字 + 3 次六爻 |
| 高级套餐 | ¥299 | 50 次八字 + 15 次六爻 |

定价页关键文案建议：

```txt
购买后次数将自动到账。若支付完成后未立即到账，请在「订单记录」中点击“刷新支付状态”或联系客服处理。
```

### 5.2 订单确认弹窗

用户点击“立即购买”后，先展示确认弹窗，而不是直接跳转支付。

弹窗内容建议：

```txt
确认购买

基础套餐
¥99

购买后获得：
- 八字对话 10 次
- 六爻解卦 3 次

请选择支付方式：
[微信支付]
[支付宝]

[确认支付]
```

作用：

- 降低误点。
- 让用户再次确认套餐内容。
- 方便后续加入优惠券、余额、折扣码。

### 5.3 微信 PC 扫码支付页

路径建议：

```txt
/payment/wechat/[paymentNo]
```

页面内容：

```txt
微信支付

请使用微信扫码支付

[二维码]

金额：¥99
套餐：基础套餐
订单号：20260507123456789

二维码将在 15 分钟后失效。
支付成功后页面将自动跳转。
```

注意：

- 前端需要轮询订单状态。
- 不要让用户无限等待。
- 超时后提示重新支付。

### 5.4 支付宝跳转页

支付宝 PC 支付通常是跳转到支付宝收银台。

页面内容：

```txt
正在跳转支付宝收银台...

如果没有自动跳转，请点击下方按钮继续支付。

[继续支付]
```

注意：

- return_url 只是同步跳回地址。
- 不得因为用户跳回就认定支付成功。

### 5.5 支付处理中页

路径建议：

```txt
/payment/result?order_no=xxx
```

页面内容：

```txt
正在确认支付结果...
请勿关闭页面。
```

前端应调用：

```txt
GET /api/orders/{order_no}/status
```

根据状态展示不同内容。

如果短时间内没有确认成功，显示：

```txt
支付结果仍在确认中。你可以稍后在「订单记录」中查看到账情况。
```

### 5.6 支付成功页

成功页要重点展示“到账权益”，不能只写“支付成功”。

页面内容建议：

```txt
支付成功

已到账：
- 八字对话 +10 次
- 六爻解卦 +3 次

当前剩余：
- 八字 18 次
- 六爻 5 次

[开始八字分析]
[开始六爻解卦]
[查看订单]
```

### 5.7 支付失败 / 取消页

页面内容建议：

```txt
支付未完成

可能原因：
- 你取消了支付
- 二维码已过期
- 支付渠道暂时不可用

[重新支付]
[返回套餐页]
[联系客服]
```

### 5.8 次数不足提示

当八字或六爻次数不足时，应展示对应服务的不足提示。

八字次数不足：

```txt
八字对话次数已用完

你可以购买基础套餐继续使用：
¥99 = 10 次八字 + 3 次六爻

[立即购买]
```

六爻次数不足：

```txt
六爻解卦次数已用完

你可以购买基础套餐继续使用：
¥99 = 10 次八字 + 3 次六爻

[立即购买]
```

注意不要笼统写“次数已用完”。用户需要知道是哪一种次数用完。

### 5.9 订单记录页

路径建议：

```txt
/orders
```

字段建议：

| 字段 | 说明 |
|---|---|
| 订单号 | 用户和客服排查使用 |
| 套餐名称 | 购买的商品 |
| 支付金额 | 实际支付金额 |
| 支付方式 | 微信 / 支付宝 |
| 订单状态 | 待支付 / 已支付 / 已关闭 / 已退款 |
| 支付时间 | 渠道确认支付时间 |
| 到账权益 | 八字 + 六爻次数 |
| 操作 | 查看详情 / 刷新支付状态 / 联系客服 |

## 六、后端系统设计

### 6.1 后端核心原则

真实支付后端必须遵守以下原则：

1. 前端不能决定支付是否成功。
2. 支付成功必须以后端验签后的异步通知或主动查单为准。
3. 金额必须以后端商品表为准，不能信任前端传入金额。
4. 支付回调必须幂等，重复回调不能重复发放权益。
5. 一个订单可以产生多条支付流水。
6. 权益发放必须和订单状态绑定。
7. 所有回调原始数据必须记录，方便排查。
8. 生产密钥不得进入 Git 仓库。

## 七、数据库设计建议

### 7.1 products 商品表

用于配置可购买套餐。

建议字段：

```txt
id
code
name
description
price_cents
currency
quota_amount
bazi_quota
liuyao_quota
active
created_at
updated_at
```

说明：

- `quota_amount` 保留用于兼容旧商品。
- 新套餐优先使用 `bazi_quota` 和 `liuyao_quota`。
- 后续权益类型超过 3 种时，应升级为商品权益明细表。

### 7.2 product_entitlements 商品权益明细表（后续推荐）

当前阶段可以先不做，但长期建议增加：

```txt
id
product_id
quota_type
amount
created_at
updated_at
```

示例：

| product_code | quota_type | amount |
|---|---|---:|
| basic_combo | chat | 10 |
| basic_combo | liuyao_chat | 3 |
| premium_combo | chat | 50 |
| premium_combo | liuyao_chat | 15 |

当后续加入风水、报告、高级模型额度、会员权益时，这种结构比不断加字段更合理。

### 7.3 orders 订单表

订单代表用户购买某个套餐这件事。

建议字段：

```txt
id
order_no
user_id
product_id
amount_cents
currency
status
created_at
paid_at
closed_at
expired_at
```

订单状态建议：

```txt
CREATED       待支付
PAYING        支付中
PAID          已支付
CLOSED        已关闭
FAILED        支付失败
REFUNDING     退款中
REFUNDED      已退款
PARTIAL_REFUNDED 部分退款
```

### 7.4 payments 支付流水表

支付流水代表某一次向微信或支付宝发起支付。

建议字段：

```txt
id
payment_no
order_id
channel
channel_trade_no
out_trade_no
amount_cents
status
prepay_payload
callback_payload
paid_at
created_at
updated_at
```

支付渠道建议：

```txt
SIMULATE
WECHAT_NATIVE
WECHAT_H5
WECHAT_JSAPI
ALIPAY_PC
ALIPAY_WAP
```

支付状态建议：

```txt
CREATED
PAYING
PAID
FAILED
CLOSED
REFUNDED
```

一个订单可以有多个支付流水。例如：用户先选择微信未支付，随后改用支付宝支付。

### 7.5 entitlements 权益发放表

用于记录用户购买后获得了什么权益。

建议字段：

```txt
id
user_id
order_id
product_id
quota_type
amount
status
granted_at
created_at
```

建议增加唯一约束：

```txt
unique(order_id, quota_type)
```

作用：防止重复回调导致重复发放权益。

### 7.6 payment_callbacks 支付回调日志表

用于记录微信/支付宝发送的异步通知。

建议字段：

```txt
id

headers
raw_body
parsed_body
signature_valid

channel_trade_no
process_status
error_message
created_at
```

这张表非常重要。没有回调日志，真实支付出问题时很难排查。

### 7.7 refunds 退款表

```txt

payment_id
channel
amount_cents
reason
status
channel_refund_no
requested_at
succeeded_at
created_at
updated_at
```

## 八、后端接口设计

### 8.1 获取商品列表

```txt
GET /api/products
```

返回示例：

```json
[
  {
    "code": "basic_combo",
    "name": "基础套餐",
    "price_cents": 9900,
    "currency": "CNY",
    "bazi_quota": 10,
    "liuyao_quota": 3,
    "description": "10次八字 + 3次六爻"
  }
]
```

### 8.2 创建订单

```txt
POST /api/orders
```

请求：

```json
{
  "product_code": "basic_combo"
}
```

返回：

```json
{
  "order_no": "20260507123456789",
  "product_name": "基础套餐",
  "amount_cents": 9900,
  "currency": "CNY",
  "status": "CREATED",
  "expired_at": "2026-05-07T12:30:00Z"
}
```

注意：

- 创建订单不代表支付成功。
- 金额必须由后端查询商品表得出。
- 前端不得传入最终支付金额。

### 8.3 创建支付

```txt
POST /api/payments/prepay
```

请求：

```json
{
  "order_no": "20260507123456789",
  "channel": "WECHAT_NATIVE"
}
```

微信 Native 返回：

```json
{
  "payment_no": "pay_xxx",
  "channel": "WECHAT_NATIVE",
  "code_url": "weixin://wxpay/...",
  "expires_at": "2026-05-07T12:30:00Z"
}
```

支付宝 PC 返回：

```json
{
  "payment_no": "pay_xxx",
  "channel": "ALIPAY_PC",
  "pay_url": "https://..."
}
```

微信 H5 返回：

```json
{
  "payment_no": "pay_xxx",
  "channel": "WECHAT_H5",
  "h5_url": "https://..."
}
```

### 8.4 查询订单状态

```txt
GET /api/orders/{order_no}/status
```

返回：

```json
{
  "order_no": "20260507123456789",
  "status": "PAID",
  "paid_at": "2026-05-07T12:20:00Z",
  "granted": {
    "bazi": 10,
    "liuyao": 3
  },
  "quotas": {
    "chat": {
      "remaining": 10
    },
    "liuyao_chat": {
      "remaining": 3
    }
  }
}
```

前端支付结果页应通过该接口轮询。

### 8.5 微信支付回调

```txt
POST /api/webhooks/wechatpay
```

处理流程：

1. 读取原始请求体。
2. 读取微信支付请求头。
3. 验证签名。
4. 解密回调 resource。
5. 校验商户号。
6. 校验 APPID。
7. 校验 out_trade_no。
8. 校验金额。
9. 查询本地 payment 和 order。
10. 判断是否已处理过。
11. 标记 payment 为 PAID。
12. 标记 order 为 PAID。
13. 调用权益发放函数。
14. 记录回调日志。
15. 返回微信要求的成功响应。

### 8.6 支付宝支付回调

```txt
POST /api/webhooks/alipay
```

处理流程：

1. 读取 POST form 参数。
2. 使用支付宝公钥验签。
3. 校验 app_id。
4. 校验 out_trade_no。
5. 校验 total_amount。
6. 校验 seller_id 或商户收款信息。
7. 判断 trade_status 是否支付成功。
8. 查询本地 payment 和 order。
9. 判断是否已处理过。
10. 标记 payment 为 PAID。
11. 标记 order 为 PAID。
12. 调用权益发放函数。
13. 记录回调日志。
14. 返回 `success`。

### 8.7 主动查单

```txt
POST /api/payments/{payment_no}/query
```

用途：

- 用户支付后未到账。
- 异步回调丢失。
- 回调处理失败。
- 后台人工排查。

查单确认支付成功后，也应走同一套支付成功处理逻辑，不能单独写一套发权益逻辑。

### 8.8 退款接口

第一版可以只做后台接口：

```txt
POST /api/orders/{order_no}/refund
```

退款前必须判断：

1. 订单是否已支付。
2. 是否已发放权益。
3. 权益是否已使用。
4. 是否允许退款。
5. 是否部分退款。
6. 是否已超过退款期限。

## 九、权益发放设计

真实支付成功后，必须统一调用同一个权益发放函数。

建议函数：

```python
def grant_product_quota(db, user_id: int, product: Product, source: str = "purchase") -> dict:
    granted = {}

    if product.bazi_quota > 0:
        QuotaService.add_quota(db, user_id, product.bazi_quota, "chat", source)
        granted["bazi"] = product.bazi_quota

    if product.liuyao_quota > 0:
        QuotaService.add_quota(db, user_id, product.liuyao_quota, "liuyao_chat", source)
        granted["liuyao"] = product.liuyao_quota

    if not granted and product.quota_amount > 0:
        QuotaService.add_quota(db, user_id, product.quota_amount, "chat", source)
        granted["bazi"] = product.quota_amount

    return granted
```

关键要求：

- 模拟支付调用它。
- 微信回调调用它。
- 支付宝回调调用它。
- 主动查单确认成功后也调用它。
- 手动补发权益也应复用它。

不要在不同支付渠道里分别写发放逻辑。

## 十、真实支付完整流程

### 10.1 标准支付流程

```txt
用户点击立即购买
→ 前端请求 POST /api/orders
→ 后端创建订单 CREATED
→ 前端请求 POST /api/payments/prepay
→ 后端创建支付流水 PAYING
→ 后端请求微信/支付宝下单接口
→ 返回二维码或跳转地址
→ 用户完成支付
→ 微信/支付宝异步通知后端
→ 后端验签、校验金额、幂等处理
→ payment = PAID
→ order = PAID
→ 发放套餐权益
→ 增加用户次数
→ 前端轮询订单状态
→ 显示支付成功和到账次数
```

### 10.2 支付宝流程重点

支付宝有两个地址：

```txt
return_url：用户支付后浏览器跳回地址
notify_url：支付宝服务器异步通知地址
```

必须注意：

- return_url 不能作为支付成功依据。
- notify_url 验签成功才是可靠依据。
- 如果 notify_url 没收到，可以主动查单。

### 10.3 微信流程重点

微信支付回调需要：

1. 验签。
2. 解密 resource。
3. 校验金额。
4. 校验商户号。
5. 校验订单号。
6. 幂等处理。

不得直接相信回调 body 里的状态字段。

## 十一、安全与幂等要求

### 11.1 回调必须幂等

微信和支付宝都可能重复发送异步通知，因此同一个订单只能发放一次权益。

建议处理逻辑：

```python
with db.transaction():
    payment = get_payment_for_update(out_trade_no)

    if payment.status == "PAID":
        return success_response()

    verify_signature()
    verify_amount()
    verify_channel_trade_no()

    payment.status = "PAID"
    order.status = "PAID"

    if not entitlement_exists(order.id):
        grant_product_quota(...)
```

必须配合：

- 数据库事务。
- 行锁。
- 唯一约束。
- 回调日志。

### 11.2 金额必须以后端为准

前端只能传商品 code，不能传最终金额。

错误设计：

```json
{
  "product_code": "basic_combo",
  "amount": 1
}
```

正确设计：

```json
{
  "product_code": "basic_combo"
}
```

后端根据商品表读取真实价格。

### 11.3 回调接口不能要求用户登录

支付回调来自微信或支付宝服务器，不是用户浏览器。

因此：

```txt
/api/webhooks/wechatpay
/api/webhooks/alipay
```

不能加用户登录校验。

但必须验签。

### 11.4 前端不能决定支付成功

错误做法：

```txt
用户跳回 success 页面 → 前端调用 add_quota
```

正确做法：

```txt
用户跳回 result 页面 → 前端查询后端订单状态 → 后端确认已支付 → 展示成功
```

### 11.5 订单号必须唯一

建议订单号格式：

```txt
年月日时分秒 + 用户标识 + 随机数
```

示例：

```txt
20260507153045U1234R839201
```

## 十四、退款规则设计

网站销售的是次数权益，因此退款规则必须提前定义。

建议第一版退款规则：

```txt
1. 支付成功但次数未使用，可申请退款。
2. 套餐次数一经使用，原则上不支持全额退款。
3. 因系统故障导致权益未到账，可联系客服处理。
4. 退款到账时间以微信或支付宝渠道处理时间为准。
```

后端需要记录：

1. 用户购买了多少次数。
2. 已经使用多少次数。
3. 剩余多少次数。
4. 是否允许退款。
5. 退款金额。
6. 退款原因。
7. 退款渠道单号。

## 十五、对账设计

真实支付上线后，每天至少需要做一次对账。

对账内容：

1. 本地 PAID 订单是否都存在于微信/支付宝账单中。
2. 微信/支付宝成功交易是否都在本地标记为 PAID。
3. 金额是否一致。
4. 订单号是否一致。
5. 退款状态是否一致。
6. 是否存在渠道成功但本地未发权益的订单。

第一版可以先做人工对账，但表结构和支付流水必须保留足够字段。

## 十六、推荐实施顺序

## 十八、关键风险与规避方案

### 风险 1：用户付钱后未到账

原因可能是：

- 回调丢失。
- 回调验签失败。
- 回调处理异常。
- 权益发放失败。

规避：

- 增加主动查单。
- 增加回调日志表。
- 增加订单后台。
- 支付成功和权益发放放入事务。

### 风险 2：重复发放次数

原因可能是：

- 微信/支付宝重复回调。
- 用户重复刷新。
- 后台重复补发。

规避：

- 回调幂等。
- 订单状态判断。
- entitlement 唯一约束。
- 数据库事务和行锁。

### 风险 3：金额被篡改

原因可能是：

- 前端传金额。
- 后端信任前端金额。

规避：

- 前端只传 product_code。
- 后端从 products 表读取价格。
- 回调时校验渠道支付金额。

### 风险 4：模拟支付被生产用户滥用

原因：

- 模拟支付接口没有环境限制。

规避：

- 生产环境关闭模拟支付。
- 或仅允许 admin 用户使用。

### 风险 5：微信内打开网页支付体验差

原因：

- 第一版只做 H5 或 Native，没有做 JSAPI。

规避：

- 第一版提示用户用浏览器打开。
- 第二阶段补 JSAPI。

---

## 十九、最终建议

当前阶段不要同时接入所有支付形态。最稳妥的方式是先把内部订单、支付、权益发放体系做扎实，然后逐个接入渠道。

推荐优先级：

```txt
P0：订单、支付流水、回调、幂等、权益发放、订单状态查询
P1：支付宝 PC / WAP 支付
P2：微信 Native 扫码支付
P3：微信 H5 支付
P4：微信 JSAPI 支付
P5：退款、对账、异常订单后台
```

真实支付系统的核心标准不是“能跳转支付”，而是：

```txt
用户付钱后一定能到账；
没有付钱绝不能到账；
重复通知不会重复到账；
异常情况能查、能补、能退、能对账。
```

只要围绕这个标准设计，后续接微信、支付宝、会员、优惠券、订阅、分销，都不会把系统搞乱。
